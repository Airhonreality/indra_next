Manual de Ingeniería Definitivo para Nango v2: Arquitectura de Alto Rendimiento para la Gestión de Identidades, Transporte y PersistenciaLa evolución de las arquitecturas de integración hacia modelos de alto rendimiento y agnósticos al proveedor ha alcanzado un hito crítico con el lanzamiento y consolidación de Nango v2. Este manual técnico documenta los estándares de ingeniería necesarios para implementar una infraestructura de conectividad que garantice la soberanía de la identidad, la eficiencia en el transporte de datos y la resiliencia en el ciclo de vida de las credenciales. Nango v2 no es simplemente una actualización de versión; representa un cambio de paradigma desde una gestión basada en máquinas virtuales hacia una infraestructura serverless totalmente aislada, diseñada para escalar bajo las demandas de agentes de inteligencia artificial y sistemas SaaS modernos que operan con volúmenes de datos masivos.Mapeo de Identidad Soberana y Gestión de ConexionesEn Nango v2, la gestión de identidades se ha refinado para permitir una vinculación determinista entre los usuarios de una aplicación externa (por ejemplo, residentes en una base de datos soberana como Indra) y las credenciales almacenadas en Nango. El sistema evita imponer un modelo de propiedad rígido, delegando la responsabilidad de la atribución a un sistema flexible de etiquetas que actúa como puente entre los UUID aleatorios de Nango y los identificadores de negocio del integrador.Estándares de Vinculación y el Uso de EtiquetasEl estándar de ingeniería v2 para vincular un userId externo con una conexión de Nango se basa exclusivamente en el uso de Connection Tags. Nango genera IDs de conexión únicos basados en el estándar UUID para asegurar la colisión cero a nivel global, lo que significa que estos IDs no contienen información semántica sobre el usuario u organización a la que pertenecen. Por lo tanto, para establecer una arquitectura agnóstica de alto rendimiento, el integrador debe inyectar el identificador externo en el objeto tags durante la creación de la sesión de conexión.El flujo estándar requiere que el backend de la aplicación solicite un token de sesión a través del endpoint POST /connect/sessions. En este paso, es imperativo definir etiquetas como end_user_id, organization_id o project_id. Estas etiquetas no solo sirven para la atribución inicial, sino que se convierten en el mecanismo principal para filtrar conexiones a través de la API, permitiendo, por ejemplo, listar todas las conexiones pertenecientes a una organización específica sin necesidad de mantener un mapeo manual complejo en la base de datos de origen.Determinismo en el Connection-Id vs. IDs AleatoriosUna duda recurrente en el diseño de sistemas de integración es si es posible pre-definir el connectionId en POST /connect/sessions para evitar la aleatoriedad. En Nango v2, cuando se utiliza el flujo de Connect UI (la interfaz gestionada por Nango), el servidor siempre genera un ID aleatorio para garantizar la integridad del sistema. Sin embargo, el integrador puede lograr un comportamiento determinista mediante el uso del Headless Client a través del método nango.auth(). En este modo, el desarrollador tiene la opción de proporcionar un connectionId específico, lo que permite que la conexión se identifique con una clave predecible desde su creación.A pesar de esta posibilidad, la recomendación de ingeniería para arquitecturas de alto rendimiento es aceptar el ID aleatorio de Nango y utilizar las etiquetas para la reconciliación. Esto se debe a que Nango envía un webhook de éxito (auth.creation) que incluye tanto el connectionId generado como las etiquetas originales. Este mecanismo permite que el backend persista la relación de forma asíncrona y segura, minimizando la exposición de IDs sensibles en el frontend.El Flujo de Upsert de ConexiónPara asegurar que un usuario nunca posea múltiples conexiones para un mismo proveedor —lo cual causaría ineficiencias en la sincronización y duplicidad de datos— se debe implementar un patrón de "Upsert de Conexión". Este flujo se divide en tres fases críticas que garantizan la unicidad del recurso:Verificación de Existencia: Antes de iniciar cualquier flujo de autorización, el sistema debe consultar el endpoint GET /connections utilizando filtros de etiquetas (ej: tags[end_user_id]=123) y el provider_config_key correspondiente.Evaluación de Validez: Si se identifica una conexión previa, se debe verificar su estado. Nango v2 facilita esto al comprobar automáticamente la validez del token cada vez que se recupera la conexión. Si el token es inválido o ha sido revocado, el sistema debe activar un flujo de reconexión.Sesión de Reconexión: En lugar de crear una nueva sesión de conexión desde cero, el integrador debe generar una "reconnect session" para el connectionId existente. Esto asegura que Nango sobrescriba las credenciales antiguas en el mismo registro, manteniendo la integridad de las etiquetas y la configuración previa del usuario.Este enfoque de "Upsert" lógico previene la proliferación de conexiones huérfanas y garantiza que el sistema de persistencia (Indra) siempre tenga una referencia uno-a-uno con el almacén de credenciales de Nango.Protocolo de Transporte: Nango Proxy v2El Proxy de Nango v2 se posiciona como una capa de transporte transparente que inyecta autenticación en tiempo real, manejando las complejidades de los protocolos OAuth y API Keys sin que las credenciales toquen el código de la aplicación del cliente. Para una arquitectura de alto rendimiento, el uso correcto del Proxy es fundamental para centralizar la observabilidad y el manejo de errores.Esquema de URL y EnrutamientoEl esquema exacto de URL para las peticiones a través del Proxy v2 sigue un modelo de ruta directa. A diferencia de otros sistemas que utilizan parámetros de consulta para definir el destino, Nango v2 utiliza el segmento de ruta {anyPath} para determinar el endpoint final en el proveedor.La estructura base es: https://api.nango.dev/proxy/{anyPath}.Cuando un desarrollador desea realizar una petición a la API de Microsoft Graph, por ejemplo al endpoint /v1.0/me, la petición al proxy debe dirigirse a https://api.nango.dev/proxy/v1.0/me. El Proxy se encarga de extraer /v1.0/me, concatenarlo con la URL base configurada para el proveedor (definida en el archivo de integración o mediante un override) y ejecutar la llamada con las credenciales adecuadas.Cabeceras Obligatorias y Sensibilidad de CaracteresEl éxito de una petición Proxy depende de la inclusión de cabeceras de configuración específicas. Por defecto, las peticiones Proxy devuelven la respuesta del proveedor "exactamente como Nango la recibe", lo que incluye el código de estado, las cabeceras y el cuerpo. Esto significa que la paginación automática no ocurre dentro del Proxy de forma nativa para peticiones síncronas; el desarrollador debe implementar la lógica de seguimiento de cursores en su propio código.Para mitigar los bloqueos por límites de tasa en silos de gran escala (Google Drive, Notion), se recomiendan dos estrategias de ingeniería:Uso de Cabeceras de Reintento: El Proxy permite el uso de la cabecera Retries. Al configurarla, Nango detectará automáticamente errores 429 (Too Many Requests) y 5xx, ejecutando reintentos con un algoritmo de retroceso exponencial que respeta las ventanas de tiempo del proveedor.Transición a Sync Functions: Para la extracción masiva de datos (bulk syncs), el Proxy síncrono no es la herramienta adecuada. En su lugar, se deben desplegar Nango Sync Functions. Estas funciones operan en un entorno serverless que soporta ejecuciones de hasta 24 horas y manejan la paginación de forma nativa mediante el uso de cursores y puntos de control (checkpoints), asegurando una ingesta de datos fluida y resistente a fallos de red o de límites de cuota.Ciclo de Vida y Resiliencia: Handshake y RefreshLa robustez de una integración se mide por su capacidad de mantener la conectividad sin intervención del usuario final. Nango v2 introduce mejoras significativas en la gestión automática del ciclo de vida de los tokens.Protocolo Estándar de Refresco de TokensUna de las ambigüedades más comunes es si el refresco de tokens requiere una intervención manual. En la arquitectura v2, el refresco de tokens es 100% automático para todas las peticiones realizadas a través del Proxy o cuando se solicita la conexión mediante la API de backend.El mecanismo funciona de la siguiente manera: antes de enviar cualquier petición al proveedor, el servidor de Nango verifica el campo expires_at del token de acceso. Si el token ha expirado o está cerca de expirar, Nango ejecuta un flujo de intercambio de refresh_token de forma síncrona. Solo después de obtener un nuevo token válido, procede a completar la petición original. Este diseño elimina la necesidad de capturar errores de invalid_credentials y manejar el refresco en la lógica de la aplicación del cliente.Sin embargo, si el refresh_token mismo ha sido invalidado (por revocación del usuario o expiración de la sesión de refresco), el Proxy devolverá un error. En este escenario extremo, es necesario implementar un catch que dispare una sesión de reconexión en el frontend, guiando al usuario de nuevo por el flujo de autorización para restaurar el acceso.Estructura de Respuesta del SDK de Frontend (onEvent)El SDK de frontend (@nangohq/frontend) es el encargado de notificar a la aplicación sobre el estado del flujo de autorización. El callback onEvent recibe un objeto que detalla el resultado de la operación. Es fundamental para el ingeniero de sistemas conocer la estructura de este objeto para la persistencia correcta de los identificadores.Para el evento de éxito (connect), la estructura es la siguiente:JSON{
  "type": "connect",
  "payload": {
    "connectionId": "<CONNECTION-ID>",
    "providerConfigKey": "<INTEGRATION-ID>",
    "provider": "<PROVIDER_NAME>"
  }
}
El connectionId reside dentro del objeto payload. Este es el valor que debe ser enviado al backend para ser persistido junto con la identidad soberana del usuario en la base de datos de Indra.En caso de error, el objeto onEvent proporciona códigos específicos como blocked_by_browser, missing_credentials o window_closed. Esta granularidad permite mostrar mensajes de retroalimentación precisos al usuario, mejorando la tasa de conversión de la integración.Especificaciones por Silo: Gestión de la Entropía del EcosistemaDiferentes proveedores de datos imponen requisitos técnicos dispares. Nango v2 actúa como un normalizador, pero existen detalles específicos para Google Drive, Notion y Google Calendar que deben ser atendidos rigurosamente.Google Drive: Scopes y Resumable UploadsGoogle Drive separa estrictamente la gestión de metadatos de la descarga de contenido, y su API de carga es altamente sensible al tamaño de los archivos.Alcances (Scopes): Para una arquitectura de alto rendimiento que solo necesita inventariar archivos, se debe utilizar el scope https://www.googleapis.com/auth/drive.metadata. Este alcance es menos sensible y suele evitar revisiones de seguridad de Google más estrictas. Sin embargo, si el sistema requiere descargar archivos o leer su contenido, es imperativo solicitar https://www.googleapis.com/auth/drive.readonly o https://www.googleapis.com/auth/drive.Protocolo de Resumable Uploads: Google recomienda el uso de "Resumable Uploads" para cualquier archivo mayor a 5 MB o para conexiones inestables. Este protocolo se puede ejecutar a través del Proxy de Nango siguiendo tres pasos:Inicio: Se envía una petición POST al endpoint de carga de Google con uploadType=resumable. Nango inyecta la autenticación y devuelve la cabecera Location de Google que contiene el "Session URI".Carga de Datos: El cliente realiza peticiones PUT directamente al "Session URI" (que ya incluye el token de sesión de Google) con los fragmentos del archivo.Monitoreo: El estado de la carga se gestiona mediante cabeceras Content-Range. Si la conexión se interrumpe, se puede consultar el progreso enviando una petición PUT vacía para obtener el último byte recibido.Notion: Notion-Version y PaginaciónLa API de Notion es conocida por su rigidez en el versionado y sus estrictos límites de tasa de 3 peticiones por segundo, lo que representa un "muro" para integraciones de nivel empresarial.Cabecera Notion-Version: Cada petición enviada a través del Proxy de Nango hacia Notion debe incluir la cabecera Notion-Version (ej: 2022-06-28). Si esta cabecera falta o es incorrecta, la API de Notion rechazará la petición o devolverá datos en un formato incompatible.Límites de Paginación: Al listar bases de datos, Notion devuelve un máximo de 100 registros por página. El sistema debe capturar el valor next_cursor de la respuesta y enviarlo en la siguiente petición como start_cursor. Es vital asegurar que el page_size no exceda los 100 ítems, ya que Notion ignorará valores superiores y podría causar inconsistencias en la recuperación de datos masivos.Google Calendar: Sincronización IncrementalPara sincronizar calendarios de forma eficiente, Google utiliza un sistema basado en tokens de sincronización que Nango v2 puede gestionar a través de sus funciones de sincronización periódica.Mecanismo Incremental: En la primera sincronización (full sync), el sistema recupera todos los eventos y recibe un syncToken. Para todas las llamadas subsiguientes, se debe proporcionar este token en el parámetro syncToken. La API solo devolverá los eventos creados, modificados o eliminados desde esa marca de tiempo.Webhooks vs. Polling: Aunque Google Calendar soporta notificaciones push (webhooks) para avisar de cambios en tiempo real, la ingeniería de alto rendimiento recomienda un enfoque híbrido. Nango v2 permite configurar webhooks que disparan una ejecución inmediata de la función de sincronización, la cual utiliza el syncToken para realizar una actualización parcial. Este modelo combina la inmediatez de los webhooks con la fiabilidad del polling incremental, garantizando que nunca se pierdan eventos debido a fallos en la entrega de notificaciones push.Resumen Técnico y Referencia de IngenieríaPara facilitar la implementación, se consolidan a continuación los recursos técnicos críticos, incluyendo los esquemas de datos y las tablas de cambios estructurales.Tabla de "Breaking Changes" Nango v1 a v2La transición a v2 introdujo cambios que impactan directamente en el mantenimiento de las integraciones existentes.Área de CambioNango v1Nango v2Acción de IngenieríaAPI Endpoints/connection (singular)/connections (plural)Actualizar todas las llamadas a API de backend.Auth KeysSecret Key única (full access)Scoped API Keys (RBAC)Crear llaves con scopes específicos (ej: proxy).ImportaciónCredenciales en la raíz del JSONCredenciales anidadas en objeto credentialsModificar los scripts de migración de datos.Configuraciónnango.yamlScripts basados en TypeScriptMigrar configuraciones a funciones Nango.Sincronizaciónnango.lastSyncDateCheckpoints (objetos flexibles)Implementar saveCheckpoint() en funciones de sync.InfraestructuraInstancias VM por clienteEjecución Serverless (AWS Lambda)Mayor rendimiento y aislamiento, sin cambio en código.JSON Schemas para Endpoints CríticosPOST /connect/sessionsUtilizado para generar tokens de sesión seguros para el frontend.JSON{

Diagrama de Flujo: La "Identidad Única" en Nango v2El flujo de datos para garantizar la soberanía de la identidad desde la infraestructura del cliente hasta el proveedor final se describe en el siguiente proceso secuencial:Origen (Indra DB): El sistema identifica que un usuario (ej: user_789) desea conectar su cuenta de Notion.Preparación (Backend): El servidor del cliente llama a Nango POST /connect/sessions incluyendo tags: { "end_user_id": "user_789" }.Tokenización: Nango devuelve un sessionToken efímero.Handshake (Frontend): El usuario completa la autorización OAuth a través de la Connect UI de Nango.Creación de Recurso: Nango genera internamente un connectionId único (UUID) y lo asocia permanentemente a la etiqueta end_user_id: user_789.Notificación (Webhook): Nango envía un webhook auth.creation al backend del cliente. El payload contiene el nuevo connectionId y la etiqueta user_789.Persistencia (Indra DB): El backend recibe el webhook, identifica al usuario user_789 y almacena el connectionId de Nango en su registro.Operación (Proxy): Para cualquier petición futura, el backend de Indra simplemente envía el connectionId guardado. Nango utiliza este ID para buscar el token y las etiquetas correspondientes, ejecutando la llamada hacia la API de Notion con la identidad correcta.Conclusión sobre la Arquitectura de Alto RendimientoLa arquitectura de Nango v2, al transicionar a un modelo serverless y basar su lógica de atribución en etiquetas, permite construir sistemas de integración extremadamente robustos y escalables. La clave para una ingeniería exitosa reside en tratar el connectionId como un recurso efímero que se mapea dinámicamente a la identidad soberana del usuario mediante etiquetas, y en aprovechar la automatización del Proxy para el transporte y el refresco de credenciales.Al seguir este manual, los ingenieros pueden asegurar que sus aplicaciones manejen el ciclo de vida de las conexiones con resiliencia, optimicen el tráfico hacia silos complejos como Google Drive o Notion, y mantengan una base de código limpia y agnóstica a las peculiaridades de cada proveedor de API. Nango v2 se establece así no solo como un gestor de autenticación, sino como el plano de control definitivo para la conectividad de datos en la era de los agentes inteligentes y el procesamiento masivo de información.
